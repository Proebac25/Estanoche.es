# Correcciones de bugs — 21/02/2026

## 1. Formulario de registro bloqueado en silencio
**Archivo:** `src/pages/auth/AltaUsuario.jsx`  
**Causa:** El navegador aplicaba validación nativa HTML5 y bloqueaba el `onSubmit` sin ningún mensaje visible al usuario.  
**Fix:** Añadido `noValidate` al elemento `<form>`.

---

## 2. Error de contraseña sin mensaje visible
**Archivo:** `src/pages/auth/AltaUsuario.jsx`  
**Causa:** `validarPassword()` actualizaba `validation.password` pero el `handleSubmit` chequaba `errors.password`, que nunca se rellenaba.  
**Fix:** Añadida validación de contraseña dentro de `newErrors` en `handleSubmit` para que el error se muestre correctamente.

---

## 3. Error 500 al verificar código de registro
**Archivo:** `api/verify-code.js` (Vercel serverless)  
**Causa:** Scripts de depuración dejaron usuarios fantasma en `auth.users` o `public.usuarios` causando colisión al crear el usuario en Supabase Auth.  
**Fix:** Limpieza manual de registros huérfanos vía SQL en Supabase.

---

## 4. Botón "Reenviar código" bloqueado al expirar el timer
**Archivo:** `src/pages/auth/VerificacionEmail.jsx`  
**Causa:** La condición `disabled={reenviando || tiempoRestante <= 0}` bloqueaba el botón exactamente cuando el usuario más lo necesitaba (al expirar).  
**Fix:** Cambiado a `disabled={reenviando}` únicamente.

---

## 5. "Cannot coerce the result to a single JSON object" al guardar ficha
**Archivos:** `src/pages/cliente/FichaC.jsx`, `src/pages/promotor/FichaP.jsx`  
**Causa:** Las fichas hacían un `UPDATE ... .select().single()` directamente con el cliente anónimo de Supabase. La política RLS devolvía un resultado incompatible con `.single()`.  
**Fix:**
- Eliminado `.single()` en ambos archivos.
- Redirigido el guardado a través del endpoint `/api/update-profile` (usa `service_role` key, bypassa RLS).

---

## 6. Botón de formulario de registro no disparaba `handleSubmit` con "Soy Promotor" activo
**Archivo:** `src/pages/auth/AltaUsuario.jsx`  
**Causa:** Misma causa que bug #1 (validación nativa HTML5). Con el checkbox de promotor, el campo de teléfono se volvía "requerido" a ojos del navegador pero sin mensaje.  
**Fix:** Cubierto por el fix del bug #1 (`noValidate`).
